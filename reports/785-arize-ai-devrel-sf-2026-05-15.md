## F — Interview Prep (STAR+R)

**STAR 1 — LLM evaluation and observability (the core DevRel story for Arize):**
- **Situation:** The Agentic Healthcare Claims Pipeline processed insurance claims using a multi-step RAG system. After initial deployment, precision was inconsistent across claim categories — some claim types were retrieved correctly >80% of the time; others fell below 50%. The root cause was unclear without systematic tracing.
- **Task:** Diagnose retrieval failures across claim types, identify whether the problem was chunking, embedding model choice, re-ranking, or prompt design, and implement a fix — all without an LLM observability tool like Arize Phoenix.
- **Action:** Manually instrumented each pipeline step with Python logging, built a custom evaluation harness that logged retrieval candidates, re-ranking scores, and final LLM responses for each test case, then analyzed patterns across failure modes. This took approximately two weeks of instrumentation and analysis before the root cause (chunking boundary misalignment for long claim documents) was identified.
- **Result:** After fixing the chunking strategy and adding cross-encoder re-ranking, precision improved ~35% and hallucination rate dropped >30%. The manual evaluation harness became a permanent fixture of the CI pipeline for regression testing.
- **Reflection:** This process — two weeks of manual instrumentation to find a root cause that Arize Phoenix would surface in minutes — is exactly the DevRel story. I'd use this as the basis for a tutorial: "How I debugged a RAG system the hard way, and what I'd use now." The personal experience with the problem makes the tooling explanation authentic, not theoretical.

**STAR 2 — Shipping public developer tooling (maps to OSS/content DevRel expectations):**
- **Situation:** The Manga Lens Chrome extension needed to abstract over multiple AI provider APIs (OpenAI, Anthropic, Google) so users could switch providers without changing their workflow. There was no existing open-source abstraction layer for this in the Chrome extension ecosystem.
- **Task:** Design and implement a provider-agnostic integration layer, ship it to the Chrome Web Store, and maintain it as a public artifact that other developers could fork or reference.
- **Action:** Built the abstraction layer in TypeScript, wrote provider-specific adapters for each API, managed credential storage in Chrome's secure storage API, and published the extension with a public GitHub repo and Chrome Web Store listing. Added documentation for each provider's setup steps.
- **Result:** Extension shipped to the Chrome Web Store and received user adoption. The multi-provider support is the primary cited differentiator in user feedback. The GitHub repo serves as a reference implementation for Chrome extensions with multi-provider LLM integration.
- **Reflection:** Shipping public code that other developers use — and writing documentation that enables them to set it up without help — is the core DevRel output loop. The Manga Lens work is my best existing example of that loop, and it's directly in the LLM tooling space that Arize serves.
