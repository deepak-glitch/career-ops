## H) Draft Application Answers (score 3.2 — included for ready-to-apply with screen flag)

**Why are you interested in this role?**
> The "Forward-Deployed AI Specialist" charter — embedding into PE portfolio companies, scoping AI use-cases, building integrations using Claude / Codex / Gemini, shipping on compressed timelines — matches what I've been doing for 2.5 yrs at an AI consulting + SaaS healthcare firm and as a solo shipper. At Progress Solutions I shipped a 5-agent healthcare claims pipeline (Intake / Validation / Consistency / Duplicate / Risk Scoring) with schema-validated JSON contracts (pydantic) between agents — audit-ready reasoning traces and HIPAA-conscious data governance. As a solo shipper I built Manga Lens — a Chrome Web Store extension with multi-provider AI vision routing across 4 providers (Claude / GPT-4o mini / GPT-4.1 Nano / Ollama) — including per-provider payload abstraction (WebP cloud / JPEG Ollama after CUDA crashes), 7-day cache, and 29 site selectors. Dream Decoder taught me that prompt engineering is infra: an intermediate structured-prompt transformation layer lifted alignment ~30% and first-pass success ~25-30%. I'm comfortable with 40% travel and excited about the cross-portfolio variety.

**Walk me through a build-vs-buy decision you've made.**
> Manga Lens multi-provider routing. I had Claude Sonnet (cloud, slower, higher quality), GPT-4o mini (cloud, fast, cheap), GPT-4.1 Nano (cloud, fastest, cheapest), and Ollama/minicpm-v (local, free, but CUDA-crash-prone). Build: a per-provider payload abstraction (WebP for cloud, JPEG for Ollama to avoid CUDA crashes), a 7-day translation cache, and per-domain selector configs for 29 sites. Buy: each provider's API as-is, plus Ollama runtime for local. Optimize routing later: cost-quality-latency trade-offs are domain-specific, so the abstraction lets me retune without re-architecting. The lesson: build the routing layer once, integrate the providers as commodities, optimize routing per use-case as data lands.
